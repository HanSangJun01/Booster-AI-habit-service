## 🐢 느린 곳 (성능) — 핵심 결과

- **데이터가 쌓여도 서버 성능은 그대로 좋음.** Prometheus 서버측 max: dashboard 0.23초, coins 0.17초. 읽기 p95 52ms.
- 즉 **"데이터 많으면 dashboard/coins가 느려질 것"이라는 1·2차의 가설은 이번 데이터 규모(25일/300건)에선 사실이 아님.** 인덱스(`user_id, check_in_date` / `user_id, created_at DESC`)가 잘 처리.
- ⇒ 개선 후보였던 **"dashboard 쿼리 5→3"은 성능 근거로는 불필요**(이미 충분히 빠름). 가독성/왕복 최소화 차원의 선택적 리팩터로만 의미.

## 💥 터지는 곳 (오류)

| ID | 엔드포인트 | 증상 | 빈도 | 성격 |
|----|-----------|------|------|------|
| F1(재현) | GET /api/dashboard/home | 연결 레벨 실패 + ~31초 stall 1건 | 3 / 37,726 (0.008%) | **앱 아님 — 연결 수락 레벨(부하도구↔서버)** |

- **검증(Prometheus):** dashboard/home 서버측 max = **0.23초**, 5xx = **0건**. 만약 DB 커넥션 풀 대기였다면 서버가 요청을 ~30초 붙들어 서버 max가 30초로 찍혔을 것 → **0.23초이므로 DB 풀·앱 내부 문제 아님.**
- **결론:** ~31초 stall은 100 VU 버스트 순간 **TCP 연결 수락 대기열**(Tomcat accept backlog) 또는 k6(Docker)↔host 네트워크 레벨에서 발생. 요청이 일단 서버에 닿으면 빠르게(≤0.23초) 처리됨.
- 2차·3차 모두 ~31초로 재현 → 우연 아님. 단 **빈도 0.008%, 서버 에러 0건**이라 사용자 영향은 극히 제한적.

## 🔌 자원 병목 (DB풀 / 메모리)

- DB 커넥션 풀/쿼리: **병목 신호 없음**(서버 max 0.2초대). 기본 풀(10)로도 이 부하·데이터에서 충분.
- 유일한 한계 신호 = 위 연결 수락 레벨(앱 외부).

## 3차 결론 (및 1~3차 종합)

- **A축 백엔드는 측정 범위 내에서 성능·안정성 모두 양호.** 100 VU 부하 + 데이터 대량에도 읽기 p95 50ms대, 서버 5xx 0건, 서버측 max 0.23초.
- 1·2·3차에서 떠오른 "느릴 것 같다"는 가설들(location 404, dashboard 500, 풀 고갈, 데이터 증가 시 둔화)은 **계기판(Prometheus)으로 검증하니 대부분 반증** — 실제 병목은 앱 안에 없었음. ← 모니터링을 단 핵심 가치(추측을 숫자로 거름).
- 유일하게 재현되는 신호는 **100 VU 버스트 순간의 연결 수락 레벨 stall(~31초, 0.008%)** — 앱 외부 현상이라 다음 주 코드 개선 대상은 아님. 향후 더 큰 부하를 다룰 때 인프라(Tomcat/스레드/스케일아웃) 관점에서 검토.
- **다음 주 액션:** 긴급 수정 대상 없음. 선택적 정리(코인 Slice, dashboard 쿼리 정돈)만 후보로 남김.
