## 🐢 느린 곳 (성능)

- 평상시 성능은 더 좋아짐(읽기 p95 61.67ms, max 403ms). 데이터가 적어 쿼리 부하는 아직 안 드러남.
- 단, 아래 F1의 stall 1건이 max iteration을 31.2초로 끌어올림(평균엔 안 묻혀도 꼬리(tail)에선 치명적).

## 💥 터지는 곳 (오류) — ★ 2차 핵심 발견

| ID | 엔드포인트 | 증상 | 빈도 | 성격 |
|----|-----------|------|------|------|
| F1 | GET /api/dashboard/home | 고부하(100 VU)에서 연결 레벨 실패 + ~30초 stall 1건 | 6 / 37,234 (0.016%) | **서버 500 아님(Prometheus 확인). 연결 레벨 실패 + 느린 200** |

- **검증 결과(Prometheus 확인, 2026-06-25):** `http_server_requests_seconds_count{status="500"}` → **결과 없음.** dashboard/home 메트릭은 `status="200", outcome="SUCCESS"` 단일 시리즈만 존재. **즉 서버는 500을 한 번도 내지 않았고, 응답한 dashboard 요청은 전부 200.** → 최초 "Hikari 타임아웃 → 500" 가설은 **반증됨.**
- **수정된 해석:** k6가 센 6건 실패(0.016%)는 **서버 컨트롤러까지 도달 못 한 연결(TCP) 레벨 실패**(100 VU 버스트 순간의 연결 거부/리셋)로 추정 — Micrometer에 안 잡힌 이유가 이것. 31.2초 stall 1건은 **실패가 아니라 "느리지만 결국 200으로 성공"**한 요청일 가능성이 큼(커넥션 풀 대기 추정).
- **남은 신호(약하지만 유효):** dashboard는 한 트랜잭션에서 쿼리 5번 → 커넥션을 가장 오래 점유. 기본 풀(10)이라 고부하에서 대기가 길어질 수 있음(31초 stall과 정합). 단 **이건 "느려짐"이지 "서버 에러"는 아님.**

## 2차 결론

- 셋업 보강으로 **1차 실패율 20% → 0.01%**, location/dashboard가 실제 데이터로 측정됨(2차 목적 달성).
- **서버 안정성은 양호:** Prometheus 확인 결과 500 에러 0건 — 서버가 응답한 요청은 전부 200. (최초 "dashboard 500 타임아웃" 추정은 반증됨)
- 다만 고부하(100 VU) 버스트에서 **연결 레벨 실패 6건(0.016%) + ~30초 stall 1건** 관찰. 서버 에러는 아니지만 **꼬리 지연/순간 연결 포화** 신호로 기록.
- 개선 후보(데이터 더 쌓이면 효과 검증 필요): 커넥션 풀 명시 설정 + dashboard 쿼리 축소로 커넥션 점유 단축. **단 현재는 우선순위 낮음**(500 없음, 빈도 0.016%).
- 다음(3차): SQL로 데이터 대량 시딩 후 측정 → 데이터량이 커질 때 dashboard 쿼리 부하/N+1과 풀 대기가 실제로 악화되는지 확인.
