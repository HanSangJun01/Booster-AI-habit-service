## 🐢 핵심 발견 — F2: 인증(BCrypt)이 유일한 실질 병목

| ID | 위치 | 증상 | 원인 | 성격 |
|----|------|------|------|------|
| F2 | POST /api/auth/login, /signup | 서버측 0.5~0.8초/건, p95 1.4~1.7초, 처리량 66 req/s | **BCrypt 해싱(CPU 집약)** — DB 아님 | 코드 버그 아님, 보안 비용 + 스케일 이슈 |

- **검증:** checkin(순수 쓰기) 서버측 0.16초로 빠름 → 느림은 쓰기/DB가 아님. login 서버측 0.79초 = BCrypt 검증 비용. **DB·커넥션풀·쿼리 문제 아님이 확인됨.**
- **의미:** 평상시(JWT 24h 만료라 유저는 하루 1회 로그인)엔 문제 없음. 단 **출시 피크(아침 동시 로그인, 이벤트 오픈 등)에 인증이 가장 먼저 느려짐.** 100 동시 로그인에서 이미 p95 1.7초.

## 🔧 개선 후보 (드디어 실질 항목)

> 4차에서 처음으로 "근거 있는" 개선 지점이 나옴. 단 **코드 버그를 고치는 게 아니라 용량/스케일 설계** 성격.

| 순위 | 항목 | 내용 | 주의 |
|------|------|------|------|
| ★1 | **수평 확장(스케일아웃)** | 로그인은 CPU 바운드 → 앱 인스턴스를 여러 대로 늘려 로드밸런싱. 출시 용량 산정의 핵심 | 가장 정석적 해법 |
| 2 | **BCrypt 강도 검토** | 현재 기본 strength 10. 10~12가 표준. **속도 위해 10 미만으로 낮추지 말 것**(보안 약화) | 보안-속도 트레이드오프, 신중히 |
| 3 | **인증 레이트리밋/큐** | 로그인·가입에 rate limit → 스파이크가 서버를 마비시키지 않게 보호 | 안정성용 |
| 4 | (이미 잘 됨) JWT 24h | 토큰 수명 길어 로그인 빈도 낮음 → 설계상 이미 완화됨 | 유지 |

- ✅ **읽기·쓰기-DB·커넥션풀은 손댈 것 없음**(1~4차 모두 양호).

## 4차 결론 (및 1~4차 최종 종합)

- **드디어 진짜 병목을 찾음: 인증(BCrypt) = 로그인 서버측 0.79초/건, 피크 동시성에서 p95 1.7초.** 1~3차의 읽기/쓰기-DB는 전부 양호했고, 유일한 실질 한계는 여기.
- **이건 "버그"가 아니라 "비용"이다.** BCrypt가 느린 건 보안상 의도된 것 → 코드를 고치는 게 아니라 **출시 전 용량 산정 + 수평 확장**으로 대비해야 함. BCrypt 강도를 낮춰 속도를 버는 것은 보안 약화라 금지.
- 평상시 트래픽(하루 1회 로그인, JWT 24h)엔 문제 없음. **출시 피크 대비**가 액션 포인트.
- 다음 주 권고: (1) 예상 피크 동시 로그인 수 산정 → 인스턴스 수 계획, (2) 인증 레이트리밋 검토. 앱 코드 리팩터는 불필요.
