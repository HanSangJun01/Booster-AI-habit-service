# A축 성능 측정 — 쉬운 버전 (BS-30)

> 이 문서 = `SUMMARY-종합요약.md`를 **초보자도 이해할 수 있게** 비유로 풀어 쓴 버전.
> 정확한 숫자/표는 `SUMMARY-종합요약.md`와 `FINDINGS_1차~5차`를 보면 됨.

---

## 한 줄 요약

> **"속도·안 터짐(성능)은 합격. 근데 6차에서 방법을 바꿔 파보니, 사람이 동시에 두 번 누르거나
> 여러 날 걸쳐 쓸 때 '돈(코인)·연속기록(스트릭)이 어긋나는' 진짜 버그 7개가 나왔다."**

즉 결론이 **두 개**다:
1. **성능(1~5차):** 출시해도 안 터진다. 몰릴 때 3군데 버벅임은 "서버 몇 대 쓸지" 문제(버그 아님).
2. **정합성(6차):** **반드시 고쳐야 할 로직 버그 7개 발견.** 부하 테스트로는 안 보이던 종류라 따로 팠다.

아래는 이 두 결론의 근거다.

---

## 1. 우리가 뭘 한 거냐 (비유)

새 식당을 열기 전에 **모의 손님을 잔뜩 보내보는 것**과 똑같다.

- **계기판 (Grafana / Prometheus)** = 주방에 CCTV랑 온도계 달기.
  → "어느 요리가 몇 초 걸리는지" 숫자로 보이게 함.
- **부하 테스트 (k6)** = 알바를 시켜 가짜 손님 수백 명을 동시에 보냄.
  → 진짜 손님 몰릴 때 어디서 줄이 막히는지 보는 것.

이걸 **조건 하나씩만 바꿔가며 5번** 했다 (1차~5차).

---

## 2. 5번을 왜 나눴냐

한 번에 다 바꾸면 "뭐 때문에 느린지" 알 수 없다. 그래서 **변수 하나씩만** 바꿨다.

| 회차 | 무엇을 바꿨나 | 비유 | 결과 |
|------|---------------|------|------|
| 1차 | 빈 DB에서 읽기 | 빈 식당에 손님 보내기 (기본 속도) | 빠름 ✅ |
| 2차 | 위치·체크인 데이터 채움 | 메뉴 데이터 채우고 다시 | 빠름 ✅ |
| 3차 | 데이터 대량 (30만 개) | 데이터를 산더미로 | 그래도 빠름 ✅ |
| 4차 | 가입+로그인+쓰기 | 손님이 회원가입·로그인까지 | **로그인 느림 발견** 🔴 |
| 5차 | 30분 지속 / 500명 동시 / 대용량 | 오래·많이·크게 한계 테스트 | 안 터짐, 동시성 한계 확인 |
| **6차** | **방법 바꿈: 속도 측정 → 코드 정밀검사** | 손님 속도 말고 **"계산서가 맞나"** 를 검사 | **돈·기록 어긋나는 버그 7개 발견** 🔴 |

---

## 3. 결국 찾은 "버벅이는 3곳" (핵심)

| 순위 | 어디가 | 왜 느린가 | 비유 |
|------|--------|-----------|------|
| 🔴 1 | **로그인** | 비밀번호 암호화(BCrypt)가 **일부러** 느리게 설계됨 (보안) | 금고 다이얼을 일부러 천천히 돌리게 만든 것. 보안상 정상인데, 사람 몰리면 줄이 생김 |
| 🟡 2 | **DB 커넥션풀** | 동시에 쓸 수 있는 DB 연결이 **10개뿐** | 은행 창구가 10개라 500명 오면 대기줄. 실제로 189명이 대기 중인 게 측정됨 |
| 🟢 3 | **코인내역 개수 세기** | 페이지 넘길 때마다 **전체 개수를 매번** 셈 | 책 페이지 넘길 때마다 전체 쪽수를 처음부터 다시 세는 것 |

> **가장 중요한 포인트:**
> 이 3개는 **"코드가 틀려서(버그) 반드시 고쳐야 하는 게" 아니다.** 지금도 잘 동작한다.
> 손님이 확 몰릴 때만 살짝 버벅이는 거고, 그 대비는 주로 **"서버 몇 대 쓸지 / 설정값"** 으로 한다.
>
> ⚠️ 단, **"버그는 없다"와 "코드를 안 바꿔도 된다"는 다른 말이다.**
> 반드시 고칠 버그는 없지만, **"맞게 돌아가지만 더 좋게" 바꿀 수 있는 코드(=선택적 최적화)는 있다.** ↓ 아래 참고.

---

## 3-1. 선택적 최적화 후보 (버그 아님, "더 좋게" 바꾸는 것)

> 코드를 다시 훑어서 찾은 후보. **안 해도 앱은 잘 돌아감.** "조금이라도 개선하고 싶을 때" 쓰는 목록.
> 다음 주에 **A축 원본 브랜치 `feature/BS-25-a-axis-backend`** 에서 진행 (이 측정 브랜치 아님).

**값어치 있는 것 (할 만함):**

| | 무엇 | 효과 | 난이도 | 비유 |
|--|------|------|--------|------|
| ① | 코인내역 `Page`→`Slice` | 의미O | 쉬움 | 페이지 넘길 때 전체 개수 매번 세는 걸 멈춤 |
| ② | DB 커넥션풀 크기 명시 (`application.yml`) | 의미O | 설정 한 줄 | 은행 창구를 10개→20개로 미리 지정 |
| ③ | 대시보드 홈 쿼리 5번 → 2~3번 | 의미O | 중간 | 자주 여는 화면이라 합치면 이득 |
| ⑥ | 복귀 스케줄러 N+1 제거 | 의미O | 중간 | 자동 배치가 미션 하나씩 DB 조회 → 한 번에 묶어 조회 |

> **⑥ 참고:** 앞에서 "N+1 없다"고 한 건 **사용자 화면(읽기) 기준**으로 맞다.
> 다만 **백그라운드 자동 실행(복귀 스케줄러)** 안에 반복문으로 DB를 하나씩 부르는 N+1이 하나 있다.
> 사용자 체감엔 영향 없어서 부하 테스트엔 안 잡혔지만, 정리하면 좋은 부분.

**해도 티 안 나는 것 (굳이 안 해도 됨):**

| | 무엇 | 왜 미미한가 |
|--|------|------------|
| ④ | 달력 정렬을 DB에서 처리 | 어차피 한 달 30개라 체감 차이 없음 |
| ⑤ | 위치등록 중복확인 쿼리 제거 | 위치 등록은 거의 안 일어남 |
| ⑦ | 체크인 시 중복 조회 | 하루 1번이라 무시 가능 |
| ⑧ | 복귀 성공 시 조회 4번 | 극히 드문 경우 |

## 3-2. 6차 — "계산서가 맞나" 검사에서 나온 진짜 버그 7개 🔴

> 1~5차는 **"음식이 빨리 나오나(속도)"** 만 쟀다. 근데 빠르다고 **"계산서 금액이 맞는" 건 아니다.**
> 6차는 방법을 바꿔서 **코드 자체를 정밀 검사(코드리뷰) + 함정 상황을 일부러 만들어 테스트**했다.
> 그랬더니 **속도 테스트로는 절대 안 보이던 버그 7개**가 나왔다.

**왜 1~5차가 못 잡았나 (중요):**
부하 테스트는 **서로 다른 손님 수백 명**을 보낸다. 근데 이 버그들은 **"같은 손님이 동시에 버튼 두 번"** 누르거나 **"여러 날에 걸쳐"** 일어나는 거라, 다른 손님 많이 보내는 테스트로는 구조적으로 못 잡는다. 렌즈가 달랐던 것.

**발견한 버그 7개 (비유):**

| 버그 | 무슨 일 | 비유 |
|------|---------|------|
| C2 동시 복귀 이중처리 | 복귀 버튼 동시에 2번 → 코인 **두 배 차감**, 출석 두 번 | ATM에서 동시에 출금 눌렀더니 두 번 빠짐 |
| C1 코인 어긋남 | 인증과 코인차감이 겹치면 **코인 잔액이 틀어짐** | 계산서 더하다 한 항목이 통째로 사라짐 |
| C4 동시 첫인증 | 같은 날 첫 인증 동시에 → 서버 **500 에러**(원랜 "이미 했음" 안내였어야) | 문 두 명이 동시에 밀어서 경첩 부러짐 |
| C5 스트릭 어긋남 | 인증+복귀 겹치면 **연속일수 +1이 사라짐** | 출석부 도장 두 손이 겹쳐 찍혀 한 칸 날아감 |
| C6 마감 경계 | 마감 '딱 그 순간'에 성공+실패가 **둘 다 처리** → 코인 -150 | 마감 종 치는 순간 합격·불합격 도장 둘 다 찍힘 |
| C7 스케줄러 중복 | 자동배치가 동시에 두 번 돌면 꼬임 | 알바 둘이 같은 정산을 각자 해서 충돌 |
| B2 가입 첫날 벌금 | 가입한 날 인증 안 했다고 **신규 회원에게 벌금** | 입사 첫날 지각했다고 벌금 매기는 셈 |
| B3 탈퇴 후에도 인증 | 탈퇴(비활성)했는데 토큰 살아있으면 **계속 인증·코인 적립** | 퇴사했는데 사원증으로 계속 출근 도장 |

> **핵심 포인트 2개:**
> 1. 이 7개는 **"더 좋게"가 아니라 "틀려서 반드시 고쳐야"** 하는 진짜 버그다. (성능 3군데랑 성격 다름)
> 2. 그중 **C2·C4는 우리 설계 문서(명세)에 "이렇게 막아라"라고 적혀 있었는데 코드가 안 한 것** → 변명 불가, 명백.

**+ 형/팀이 정해줘야 할 정책 3개** (명세에 안 적혀 있어 결정 필요):
- B1: "곧 끊길 스트릭"에 7일 보상 100코인을 미리 줘버리는 문제 — 보상 타이밍 어떻게?
- B2: 가입 첫날 미인증 면제할지 (그래야 맞을 듯)
- B3: 탈퇴 후 인증/코인을 어디까지 막을지

**+ 오해 1개 정정:** 처음엔 "복귀한 날도 출석으로 쳐야 하는 거 아냐?"를 버그로 봤는데, **설계 문서(Phase 3)를 보니 "복귀는 빠진 날만 메우고 오늘은 따로 인증" 이 의도된 설계**였다. → 버그 아님으로 정정.

## 4. 안심해도 되는 것 (성능/안정성 — 1~5차 기준)

- 서버가 **한 번도 안 죽음** — 5xx 에러 0건 (전 구간)
- **메모리 누수 없음** — 30분 연속 돌려도 메모리 일정 (102MB → 108MB로 거의 평평)
- **N+1 문제 없음** — DB 쿼리 비효율의 대표 문제인데, 우리 구조엔 아예 없음
- **인덱스 완벽** — 데이터 30만 개여도 조회는 0.04ms (눈 깜짝할 새)

> ⚠️ 이 "안심"은 **속도·안 터짐** 얘기다. **계산 정확성(로직)은 위 3-2의 7개 버그**가 별개로 있다. 둘을 헷갈리면 안 됨.
