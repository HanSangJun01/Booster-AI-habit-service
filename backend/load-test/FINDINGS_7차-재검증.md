## 0. 핵심 결론

- **6차 수정 8종(C1/C2/C4/C5/C6/C7 + B1/B2/B3)은 정상 + 데드락 없음** — 동시성/로직 리뷰어가 독립적으로 확인. 락 부분순서 `RecoveryMission < User < Streak` 사이클 없음.

### 보너스 실험 (이번 라운드 수정·최적화 실증)
**Exp A — 커넥션풀 A/B (500VU 동일 부하):** 최적화 ② 입증
| pool | active | pending 피크 | read p95 | 처리량 |
|------|--------|-------------|----------|--------|
| 10 (기존) | 10 | **189** (원래 5차와 일치) | 1.46s | 283 req/s |
| 30 (최적화) | 30 | **169** | **1.02s** | **430 req/s** |
→ 처리량 **+52%**, p95 **−30%**. (pending은 500VU≫풀이라 여전히 높음 → 출시 용량 산정: pool 크기 × 인스턴스 수 정합 필요.)

**Exp B — 동시성 수정 E2E (실서버 HTTP):** C4/F4 입증
- 같은 유저 동시 체크인 10건 → **201×1 + 409×9, 5xx=0**
- 같은 이메일 동시 가입 8건 → **201×1 + 409×7, 5xx=0**
→ 단위테스트뿐 아니라 실제 구동 서버에서도 이중처리/500 누출 없음.

**Exp C — 대시보드 쿼리수 (generate_statistics):** 최적화 ③ 검증
- `/api/dashboard/home` 세션당 **3 JDBC statements** (5→3 확정).

**5차 soak (30분 지속, 30VU):** 누수 검사 — read p95 **20.85ms**, 25.4만 요청 **실패 0%**, **5xx=0**. heap 55→106MB(30분 피크 112, 톱니형 안정 = 누수 아님). 원래 5차 "누수 없음" 결론과 일치.

## 3. 2단계에서 관찰한 성능 개선점(추가)
- **BCrypt 로그인/가입이 여전히 최상위 부하** (login p95 811ms@100VU). 코드 최적화 대상 아님 → **인스턴스 수평확장 + 인증 레이트리밋**(용량 산정 항목).
- **커넥션풀 pending 169@500VU** — pool=30로도 500VU는 대기 발생. 출시 시 **예상 피크 동시요청 → pool×인스턴스** 산정 필요.
- **코인 COUNT(①)** 는 30만행 기준에서만 유의(3차 p95 88ms). 실제 유저는 수십 건이라 체감 0 → ① Slice는 여전히 선택.
