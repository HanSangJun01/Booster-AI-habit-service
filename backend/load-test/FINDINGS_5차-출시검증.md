# A축 출시 전 심화 검증 보고서 — 5차 (BS-30)

> 1~4차(버스트·기능별)에 더해, "실제 출시" 관점에서 **A: 지속(soak) / B: 고동시성 / C: 대용량 쿼리플랜**을 검증.
> 목적: 더 긴 시간·더 높은 동시성·더 큰 데이터에서 숨은 병목이 있는지.
> 스크립트: `a-axis-soak-test.js`, `a-axis-stress-test.js`, `seed-big.sql`.

---

## B. 고동시성 스트레스 (VU 100→300→500)

- 결과: 처리량 344 req/s, 실패율 **0.00%**(`dial: i/o timeout` 2건만), 읽기 클라 p95 **1.15초**, iteration max **51.8초**.
- 서버측 p95(최근 5m): dashboard **0.70초**, coins **0.67초** (100 VU 땐 0.23초 → 500 VU에서 약 3배 증가).
- CPU: process(JVM) 20% / system 82%. **JVM이 CPU 바운드가 아닌데도 서버측 지연 증가** → 원인은 CPU 아닌 **동시성 한계(커넥션풀 10 / Tomcat 스레드 대기)**.

**B 결론:** 읽기 경로는 500 VU에서도 **에러 없이 "느려지기만"**(우아한 degradation). 단 **수백 동시성부터는 커넥션풀/스레드 대기**가 서버측 지연을 키움 — 100 VU에선 안 보이던 한계가 여기서 처음 드러남. 연결 수락 한계(i/o timeout, 51초 stall)도 고동시성에서 악화.

## C. 대용량 데이터 쿼리플랜 (코인내역 30만 건 + EXPLAIN ANALYZE)

| 쿼리 | 실행계획 | 실측 시간 | 평가 |
|------|---------|----------|------|
| 코인내역 페이징(LIMIT 20) | Index Scan (`idx_coin_transactions_user_created`) | **0.04ms** | ✅ 완벽 |
| 코인내역 COUNT | Parallel **Seq Scan** (인덱스 미사용) | **28ms** | ⚠️ 유일한 풀스캔 |
| 대시보드 캘린더(월 범위) | Bitmap Index Scan (`idx_personal_check_ins_user_date`) | **0.02ms** | ✅ 완벽 |

**C 결론:** 핫 조회는 30만 건에도 sub-millisecond(인덱스 완벽). **유일한 비인덱스 연산은 페이징 COUNT**(Seq Scan, 28ms@30만). 이는 4차까지 "약함"이던 **코인 Page→Slice 후보에 실제 근거**를 줌 — 단 30만 건은 비현실적 극단치라 현재는 무해, 데이터 폭증 대비 선택지.

## A. 지속(soak) 30분 — 완료 ✅ 누수 없음

- constant 30 VU로 30분 연속(197,829 요청 / 39,565 iter / 실패율 0.00%).
- **JVM 힙:** 30분간 피크 102MB→108MB로 거의 평평. 우상향 누적 없음 → **메모리 누수 없음.**
- **GC:** 30분 누적 5.3초(1280회)로 규칙적 회수. GC 압박 없음(전체 시간의 ~0.3%).
- **스레드:** 부하 중 최고 217 → 종료 후 27로 반납 → **스레드 누수 없음.**
- (주의) soak 구간 read p95=208ms는 같은 시간대에 돌린 B(500 VU)·C(30만 시드)와 겹쳐 부풀려진 값. soak의 목적은 지연이 아니라 누수 판정이며, 누수 없음으로 통과.
- **보너스 확증:** 측정 윈도우(35m, B 스트레스 포함)에서 **Hikari pending 최고 189 / active 최고 10** 관측. B에서 "커넥션풀이 병목"이라던 추정이 숫자로 확정됨(피크에 189건이 커넥션 대기). 풀=10이 고동시성에서 부족 신호.

## 종합 — 출시 관점 최종 (1~5차)

**버그/장애: 없음.** 1~5차 통틀어 서버 5xx 0건, 데이터·동시성·시간 어디서도 크래시 없음.

**실증된 "스케일 레버" 3개** (코드 버그 아닌 용량/설계):
1. **인증 BCrypt (4차)** — 가장 큰 병목. 로그인 서버측 0.79초/건. → 수평 확장 + 레이트리밋. BCrypt 강도 낮추기 금지.
2. **커넥션풀/스레드 (B+A 확증)** — 수백 동시성부터 읽기 서버측 지연 증가(0.23→0.70초). CPU 아닌 동시성 한계. **Hikari pending 최고 189 / active 10 측정**으로 풀(10) 부족 확정. → 풀/스레드 크기 출시용으로 명시 설정, 인스턴스 수 계획.
3. **코인 COUNT (C)** — 페이징 COUNT만 Seq Scan. → 총개수 불필요하면 Slice로 전환(선택, 데이터 폭증 대비).

**안심해도 되는 것:** 읽기/쓰기-DB 쿼리, 인덱스(30만 건 검증), N+1(구조적으로 없음 — JPA 연관 미사용), 500 에러.

**다음 주 권고 우선순위:** ① 출시 피크 동시 로그인 산정→인스턴스/풀 사이징(1·2번 묶어서) ② (선택) 코인 Slice. **앱 로직 리팩터 불필요.**
